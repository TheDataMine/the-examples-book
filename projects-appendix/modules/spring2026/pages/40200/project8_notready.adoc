The Ising model is a simple mathematical model that helps us understand the magnetic properties of a material. In this model, the material is represented as a grid composed of small squares, with each square containing an atom. Each atom has a property called a “spin,” which can be either “up” (+1) or “down” (−1). The spins of the atoms can change over time and depend on the temperature. Spin arrangements with lower energy, meaning they are more balanced and stable, are more likely to occur. In the beginning, spins are random, and the system is not in balance. Then, what happens? After a while, they tend to be organized into patterns that reduce the system's energy based on the temperature. 

To analyze these systems, all possible spin arrangements were collected, including those with high energy. In nature, systems usually prefer states with the lowest energy, so many of the high-energy arrangements did not represent the material's actual state. This resulted in noisy and unreliable data. Including all energy levels complicated the analysis (This is how such a study was done before the Metropolis Algorithm).

The Metropolis Algorithm helps by choosing mostly low-energy, more likely, and balanced spin arrangements to sample in the model. At each step, it suggests a new spin arrangement and decides whether to accept it based on its energy. This way, the system primarily resides in realistic, low-energy states. As a result, meaningful and physically possible examples are gathered, making the analysis easier by ignoring unnecessary and misleading arrangements.

Metropolis Algorithm is actually a type of Markov Chain Monte Carlo (MCMC) method. Although MCMC encompasses many algorithms, the Metropolis Algorithm is a classic and straightforward one that works exceptionally well for physical systems, such as the Ising model, thereby making computations easier and more effective.

=== Question 2 (2 points)
As a first step in Metropolis Algorithm, we need to derive the loglikelihood function for the model that we want to fit. The likelihood is the probability of observing the data given the model’s parameters.

*Step 1:* Derive the likelihood function from the model.

For the given model, $y = \beta_0 + \beta_1 * x + error$, where the error terms are assumed to have normal distribution. This is the required assumption to build linear regression model. The normal distribution means that most values are close to the average, and very small or very large values are rare. For instance, in a classroom, the majority of students' heights are around the average, with the number of those who differ from the average decreasing as you go to both long and short extremes. This pattern of data is what we call a normal distribution.

In our model, the error has an average of zero, meaning most predicted values are close to the real observed values, and the differences between them usually follow this normal pattern. What we want to do is take the parameters ($\beta_1$, $\beta_0$, $\sigma$) and calculate how likely it is to get the actual observed data under this model. To do this, we first calculate the difference between the model’s predicted values and the real observed values. Then, we measure how probable it is for these differences to occur according to the normal distribution. Finally, this gives us a number that tells us how well the model explains the data — in other words, how likely the data is under the model.

What is distribution?

Distributions provide more than just individual probabilities; they offer a comprehensive view of all possible outcomes and their corresponding probabilities. They serve as a detailed map of uncertainty for a specific situation. Consider distributions to be the mathematical way of expressing, “This occurs frequently, that occurs occasionally, and this other thing happens infrequently” - but with specific numbers assigned to each outcome. These distributions help us to predict the future, make decisions an understand the patterns.

Now we will write a function, called likelihood function, that tells us how likely our data is, based on some choices we make in the model. But when we calculate this directly, computers can have trouble with very small numbers. 

To fix this, we use the logarithm (log) of this function instead, which makes the calculations easier and more reliable when the algorithm runs.
[source,R]
----
likelihood <- function(beta1, beta0, sigma) {

  y_pred <- beta1*x + beta0
  ll <- sum(dnorm(y, mean = y_pred, sd = sigma, log = TRUE))
  return(ll)
}
----
This allows us to put in the data (y), covariate values (x), and values of beta1, beta0 and sigma as arguments and the function will return the log likelihood.

*Step 2:* Define the prior distributions for parameters.

Next step in Metropolis Algorithm is defining a prior distribution for each of the parameters (beta1, beta0, sigma). 

At first, we make a guess based on what we know about the slope parameter showing the relationship between Pedal$Width and Pedal$Length. Since we know that there is a positive relationship between Pedal$Width and Pedal$Length, we have to choose a distribution which generates positive values. These choices are “informative” because we have some knowledge about the parameters from the linear model. Alternatively, if you have any background information about the length and width of the sepals and petals of iris flowers, you can incorporate this knowledge into your prior distributions. In the absence of prior knowledge, selecting a prior distribution will be “uninformative”, but you should be careful about the choosing the distribution with large spread or variance.
[source,R]
----
prior <- function(beta1, beta0, sigma) {
  pp <- sum(dunif(beta1, min = 0, max = 30, log = TRUE),
            dnorm(beta0, mean = 0, sd = 5, log = TRUE),
            dunif(sigma, min = 0, max = 30, log = TRUE))
  return(pp)
}
----
The function above calculates the sum of the log-prior probabilities for the parameters based on their defined prior distributions. The default output of the `dunif` and `dnorm` functions is the probability density values, not in logarithmic scale. Since we work in logarithmic scale, we set `log=TRUE`. Notice that if you run the default code or set `log=FALSE`, the probabilities will be multiplied instead of being summed.
[source,R]
----
prior <- function(beta1, beta0, sigma) {
  pp <- (dunif(beta1, min = 0, max = 30)*
            dnorm(beta0, mean = 0, sd = 5)*
            dunif(sigma, min = 0, max = 30))
  return(pp)
}

*Step 3:* The posterior distribution

We currently have initial beliefs (prior) about the parameters. Then, we have additional information from the likelihood of the data given the parameters. Now, you combine the prior knowledge you had with this new information to make a more accurate estimate of the parameters. This is called posterior distribution.

`posterior` `~` `likelihood` `x` `prior`

You may notice that we used `~` instead of `=` since the posterior distribution is proportional to the product of the likelihood and prior distribution, but not exactly equal. Because there is a normalizing constant ensures that all possible values for the parameter values sum (or integrate) to 1- This depends on whether the random variable is discrete or continuous. We started with a prior belief, for example beta1 has unif (0, 30). And we have new information coming from the likelihood telling us how probable the data is for each possible beta1 value. The posterior combines these two, but a probability distribution must integrate to 1. The normalizing constant is the value you divide by to ensure that the total probability of all possible beta1 values equals 1. This constant is often complicated to calculate.
Mathematically, we express this as:

\[
\text{Posterior} = \frac{\text{Likelihood} \times \text{Prior}}{\text{Normalizing Constant}}
\]

[source,R]
----
# Posterior distribution
posterior <- function(beta1, beta0, sigma) {
   return(likelihood(beta1, beta0, sigma) + prior(beta1, beta0, sigma))
 }
----

=== Question 4 (2 points)
*Monte Carlo Method — A Very Simple Explanation:*

Let’s recall that we have a system, like a group of atoms or molecules, as exemplified by the Ising Model presented at the beginning of this project. This system can exist in many different “states”, and each state has a different energy. We want to calculate the average energy of the system or some other property.

Here’s how it was done before Metropolis Algorithm:

*1.	Generate Random States:* We randomly pick many possible states of the system. For example, we give atoms or molecules different positions.

*2.	Calculate the Energy:* For each state, we calculate its energy.

*3.	Identify Important States:* Not all states are equally important! Low-energy states are more likely, while high-energy states are less likely.

*4.	Assign Weights:* We assign a weight to each state based on its energy. This weight is calculated using the Boltzmann function.

*5.	Compute the Average:* Finally, we use these weights to calculate the average value of the property we are interested in.

*But This Was Very Difficult and Inefficient:*

Why? Because we were selecting states randomly. Most of the time, high-energy states (which have low probabilities) were being selected. These states have very small weights but exist in large numbers. As a result, the computation became unnecessarily complicated and inefficient.

*Metropolis Algorithm — What Changed?*

Instead of selecting random states and weighting them, let’s directly select states based on their energy-dependent probabilities!

In other words:

•	Hardly ever select states with very high energy.

•	Select low-energy states much more frequently.

This way, all selected states have equal weight, and there’s no need to assign weights anymore.

Now, for our example, we want to generate a sample for each parameter (beta1, beta0, and sigma). Initially, we considered all possible parameter values regardless of how realistic or meaningful they were.
This meant that many of the parameter values we sampled were either highly unlikely or didn’t contribute much to our understanding. However, the Metropolis Algorithm aims to sample from the target distribution, which is often the posterior distribution, representing the most probable values of the parameter given the data. At each step, the algorithm uses a proposal distribution (normal in our example) to suggest a new value for each parameter. If the proposal distribution is not symmetric, the algorithm is called Metropolis-Hasting algorithm, instead of Metropolis Algorithm. It then decides to accept or reject these values based on its likelihood or energy level. This process allows us to prioritize probable parameter values and avoid spending computational resources on unlikely or irrelevant ones.

As a result, compared to the earlier approach, the Metropolis Algorithm simplifies the sampling process, improves efficiency, and helps us gather more meaningful data about the parameter distribution.

=== Question 6 (2 points)
If the proposal distribution we select is too *narrow*, the chain may repeatedly sample the same region, hindering its exploration of the entire space. Conversely, if the proposal distribution is too *wide*, many proposed samples will be rejected due to the chain’s tendency to jump to vastly different areas. Ideally, the acceptance rate should be around 50%. However, in high-dimensional or complex problems, an acceptance rate of approximately 23% is considered acceptable (http://www.stat.columbia.edu/~gelman/research/published/theory7.ps).

Based on the information provided, we can determine the acceptance rate of our Metropolis chain to assess whether it falls within a reasonable range. We can incorporate an acceptance rate into the existing code. By maintaining the code’s structure, we can count the number of accepted steps and calculate the ratio at the end.

[source,R]
----
# Metropolis Algorithm with acceptance rate calculation
start_values <- c(1, 1, 1)  # initial values for beta1, beta0, sigma
iterations <- 100000
chain <- matrix(0, nrow = iterations + 1, ncol = 3)
colnames(chain) <- c("beta1", "beta0", "sigma")
chain[1, ] <- start_values

accepted <- 0  # counter for accepted proposals

for (i in 1:iterations) {
  # Propose new values for beta parameters using normal distributions
  beta1_proposal <- rnorm(1, mean = chain[i, 1], sd = 0.02)
  beta0_proposal <- rnorm(1, mean = chain[i, 2], sd = 0.03)
  
  # Propose new value for sigma (standard deviation) using uniform distribution
  sd_proposal <- runif(1, min = 0.01, max = chain[i, 3] * 2)
  
  proposal <- c(beta1_proposal, beta0_proposal, sd_proposal)
  
  # Calculate acceptance probability
  acceptance <- exp(
    posterior(proposal[1], proposal[2], proposal[3]) -
    posterior(chain[i, 1], chain[i, 2], chain[i, 3])
  )
  
  # Decide whether to accept the proposal
  if (runif(1) < acceptance) {
    chain[i + 1, ] <- proposal
    accepted <- accepted + 1  # proposal accepted
  } else {
    chain[i + 1, ] <- chain[i, ]  # stay at current position
  }
}

# Calculate the acceptance rate
acceptance_rate <- accepted / iterations
cat("Acceptance rate:", acceptance_rate, "\n")

----
Although the acceptance rate is low (2.5%), the trace plot appears promising. This suggests that even with a low acceptance rate, the parameters are mixing sufficiently. In high-dimensional problems or when parameters are on different scales, a low acceptance rate is normal, especially for constrained parameters like sigma. Our acceptance rate may primarily be due to the sigma updates. We could consider using a lognormal distribution instead of a uniform distribution.

sd_proposal <- rlnorm(1, meanlog = log(chain[i,3]), sdlog = 0.03)



  proposal <- c(beta1_proposal, beta0_proposal, sd_proposal)
  acceptance <- exp(posterior(proposal[1], proposal[2], proposal[3]) -
                      posterior(chain[i, 1], chain[i, 2], chain[i, 3]))
  
  if (runif(1) < acceptance) {
    chain[i + 1, ] <- proposal
    accepted <- accepted + 1 
  } else {
    chain[i + 1, ] <- chain[i, ]
  }
}
